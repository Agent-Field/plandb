Getting Started

This guide walks you through PlanDB from first init to multi-agent parallel execution. Each step builds on the last, giving you a working task graph by the end.

Initialize Project

Every PlanDB project starts with plandb init. This creates a .plandb.db SQLite database in your current directory to store tasks, dependencies, and execution state.

Terminal
$ plandb init "my-project"
 
Created project p-my-project
Database: .plandb.db
Run `plandb add` to create your first task.

The project name is used for display and filtering. You can have multiple .plandb.db files in different directories for different projects.

Add Tasks

Add tasks with a title and optional metadata. The --kind flag categorizes the work, and --priority controls ordering when multiple tasks are ready.

Terminal
$ plandb add "Design schema" --kind research --priority 10
 
Created t-001 "Design schema" [research, priority 10]
 
$ plandb add "Build API endpoints" --kind code --priority 8
 
Created t-002 "Build API endpoints" [code, priority 8]
 
$ plandb add "Write integration tests" --kind test --priority 5
 
Created t-003 "Write integration tests" [test, priority 5]

Available kinds: code, research, review, test, shell, generic. Higher priority numbers are picked up first.

Add Dependencies

Wire tasks together with --dep. A dependent task stays in pending status until all its upstream tasks are done. Dependency kinds control how tasks relate.

Terminal
$ plandb add "Build API endpoints" --dep t-001:feeds_into --kind code
 
Created t-004 "Build API endpoints" [code]
dep: t-001 (feeds_into) — blocked until t-001 completes
 
$ plandb add "Write API tests" --dep t-004:feeds_into --kind test
 
Created t-005 "Write API tests" [test]
dep: t-004 (feeds_into) — blocked until t-004 completes

Dependency types:

Kind Meaning Example
feeds_into Result data flows to dependent task Schema design feeds into API build
blocks Ordering only, no data flow Deploy must wait for tests
suggests Soft dependency, does not block Docs are nice to have before review

Check Status

See the full project state with plandb status --detail. It shows task counts by status, the dependency graph, and what is ready to work on next.

Terminal
$ plandb status --detail
 
my-project — 0/3 done
 
ready 1 pending 2 done 0
 
Ready:
t-001 Design schema [research] priority 10
 
Pending:
t-004 Build API endpoints [code] blocked by t-001
t-005 Write API tests [test] blocked by t-004

Use plandb project dag to see the full dependency tree, or plandb status --json for machine-readable output.

Start Working

Claim the next ready task with plandb go. PlanDB picks the highest-priority ready task, atomically marks it as running, and shows you what to work on including upstream context.

Terminal
$ plandb go
 
→ t-001 "Design schema" [1/3 · 1 ready · 2 blocked]
 
downstream:
t-004 "Build API endpoints" (receives YOUR result)

The atomic claim prevents two agents from grabbing the same task. The downstream section reminds you what depends on your output.

Complete & Continue

When finished, mark the task done with plandb done. The --next flag automatically claims the next available task. Use --result to pass data to downstream tasks.

Terminal
$ plandb done --result '{"schema":"users, posts, comments"}' --next
 
✓ t-001 "Design schema" completed
result stored — available to downstream tasks
 
→ t-004 "Build API endpoints" [2/3 · 1 ready · 1 blocked]
 
upstream:
t-001 → (feeds_into) result: {"schema":"users, posts, comments"}
 
downstream:
t-005 "Write API tests" (receives YOUR result)

The result from t-001 is now visible to t-004 as upstream context. This is how data flows through the task graph without shared state.

Split Tasks

When a task turns out to be larger than expected, split it into subtasks. Dependencies are automatically rewired: the parent's upstream feeds into each new subtask, and the parent's downstream waits for all of them.

Terminal
$ plandb task split t-004 --into '[
{"title": "Auth endpoints"},
{"title": "Database layer"},
{"title": "REST API routes"}
]'
 
Split t-004 "Build API endpoints" into 3 subtasks:
 
t-006 Auth endpoints ready
t-007 Database layer ready
t-008 REST API routes ready
 
deps rewired: t-001 → [t-006, t-007, t-008] → t-005
original t-004 cancelled

After splitting, all three subtasks are ready in parallel since their shared upstream (t-001) is already done. Downstream task t-005 will wait for all three to complete.

Multi-Agent Execution

PlanDB's atomic claiming makes it safe for multiple agents to work in parallel. Set PLANDB_AGENT to identify each worker. Independent ready tasks are distributed automatically with no conflicts.

Terminal — Agent 1
$ PLANDB_AGENT=worker-1 plandb go
 
→ t-006 "Auth endpoints" [2/5 · 3 ready · 1 blocked]
Terminal — Agent 2
$ PLANDB_AGENT=worker-2 plandb go
 
→ t-007 "Database layer" [2/5 · 2 ready · 1 blocked]
Terminal — Agent 3
$ PLANDB_AGENT=worker-3 plandb go
 
→ t-008 "REST API routes" [2/5 · 1 ready · 1 blocked]

Each agent atomically claims a different task. When all three finish, t-005 "Write API tests" becomes ready and the next plandb go picks it up. Use plandb task list --status running to see what each agent is working on.
