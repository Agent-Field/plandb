CLI Reference

Complete command reference for plandb. Every flag, every option, from actual --help output.

Core Workflow

The five commands you use 90% of the time: create a project, add tasks, claim work, complete it, check progress.

plandb init

Create a project and set it as default. Shortcut for plandb project create.

$ plandb init "auth-system"

Arguments

NameRequiredDescription
<NAME>YesProject name

Options

FlagTypeDescription
--descriptionstringOptional description of the project's goal

Example

$ plandb init "auth-system" --description "JWT-based authentication service"
Created project p-auth-system (set as default)

plandb add

Create a new task. Shortcut for plandb task create. The title is positional.

$ plandb add "Design API schema" --dep t-research

Arguments

NameRequiredDescription
<TITLE>YesTask title

Options

FlagTypeDefaultDescription
--projectstringcurrentProject ID (uses default if set via plandb use)
--asstringCustom short ID (e.g. --as api creates t-api)
--kindenumgenericTask kind: generic, code, research, review, test, shell
--descriptionstringDetailed description of what the task involves
--priorityint0Priority (higher = more important)
--depstringDependency: TASK_ID (default: feeds_into) or TASK_ID:KIND. Repeatable.
--parentstringParent task ID (for hierarchical decomposition)
--max-retriesint0Max auto-retry attempts on failure
--timeoutintTimeout in seconds (reclaims task if exceeded)
--requires-approvalflagRequire human approval before task completes
--tagstringTags for filtering. Repeatable: --tag api --tag auth

Example

$ plandb add "Implement auth handler" --as impl --kind code --dep t-design --tag backend
Created t-impl "Implement auth handler" [code] depends on: t-design

plandb go

Claim and start the next ready task in one command. The preferred agent entry point. Atomic: no two agents claim the same task.

$ plandb go --agent worker-1

Options

FlagTypeDefaultDescription
--agentstringdefaultAgent identifier. Also reads PLANDB_AGENT env var.
--projectstringcurrentProject ID (uses default if not set)

Example

$ plandb go --agent claude-1
--> t-impl "Implement auth handler" [2/8 - 3 ready - 1 blocked]

upstream:
  t-design --> (structured)

downstream: t-tests "Write auth tests" (receives YOUR result)

plandb done

Complete a task, optionally attach result data and claim the next ready task in one shot.

$ plandb done --result '{"api":"done"}' --next

Arguments

NameRequiredDescription
[TASK_ID]NoTask ID to complete. Omit to complete your current running task.

Options

FlagTypeDefaultDescription
--resultstringResult data (JSON or plain text). Passed to downstream tasks via handoff.
--filesstringFiles modified (comma-separated paths). Enables conflict detection.
--nextflagAfter completing, claim + start next ready task.
--agentstringdefaultAgent ID. Also reads PLANDB_AGENT env var.

Example

$ plandb done --result '{"files":["src/auth.rs"]}' --next --agent worker-1
Completed t-impl "Implement auth handler"
--> t-tests "Write auth tests" [3/8 - 2 ready]

plandb status

Show project progress: done/total, ready tasks, running agents. Three detail levels.

$ plandb status --detail

Options

FlagTypeDescription
--projectstringProject ID (uses default if not set)
--detailflagShow per-task breakdown with status icons
--fullflagShow all tasks and dependency edges

Example

$ plandb status
auth-system: 3/8 done - 2 ready - 1 running (worker-1)

$ plandb status --detail
[x] t-research "Research auth patterns"
[x] t-design   "Design API schema"
[x] t-impl     "Implement auth handler"
[>] t-tests    "Write auth tests"         (worker-1)
[o] t-review   "Code review"
[o] t-docs     "Write documentation"
[.] t-deploy   "Deploy to staging"
[.] t-e2e      "Run E2E tests"

plandb show

Show full details of a task. Shortcut for plandb task get. Supports fuzzy ID matching.

$ plandb show t-impl

Arguments

NameRequiredDescription
<TASK_ID>YesTask ID (supports fuzzy matching)

Example

$ plandb show t-impl
id: t-impl
project: p-auth-system
title: Implement auth handler
status: done
kind: code
priority: 0
description: Implement JWT-based auth handler with refresh tokens.
result: {"files":["src/auth.rs"]}

plandb list

List tasks with optional filters. Shortcut for plandb task list.

$ plandb list --status ready --kind code

Options

FlagTypeDescription
--projectstringProject ID
--statusenumFilter by status: pending, ready, claimed, running, done, failed
--kindenumFilter by task kind
--tagstringFilter by tag
--agentstringFilter by agent

Example

$ plandb list --status ready
[o] t-review "Code review"         ready
[o] t-docs   "Write documentation" ready

plandb split

Split a task into sub-tasks. Shortcut for plandb task split. Supports comma-separated titles, chain syntax with >, or JSON arrays.

$ plandb split --into "Parse tokens, Build AST, Generate IR"

Arguments

NameRequiredDescription
[TASK_ID]NoTask ID to split. Omit to split current running task.

Options

FlagTypeDescription
--intostringParts: comma-separated ("A, B, C"), chain ("A > B > C"), or JSON array

Example

# Parallel sub-tasks (no dependencies between them)
$ plandb split --into "Unit tests, Integration tests, E2E tests"

# Sequential chain (each depends on the previous)
$ plandb split --into "Parse > Validate > Transform"

Task Management

Lower-level task lifecycle commands available under plandb task.

plandb task create

Create a new task in a project. Same as plandb add but under the task subcommand.

See plandb add for the full options table.

plandb task create-batch

Create multiple tasks from a YAML file. Ideal for bootstrapping a project plan.

$ plandb task create-batch --file tasks.yaml

Options

FlagTypeDescription
--filepathYAML file with task definitions (see docs for schema). Required.
--projectstringProject ID (uses default if not set)

YAML Format

# tasks.yaml
tasks:
  - id: design
    title: "Design API schema"
    kind: research
    priority: 10
    description: |
      Research auth patterns and design the API.
  - id: impl
    title: "Implement handler"
    kind: code
    deps: [{ task: design, kind: feeds_into }]

Plan Adaptation

Mid-flight replanning: insert steps, amend context, pivot subtrees, split tasks, or replan entirely.

plandb task insert

Insert a new task between two existing tasks, rewiring dependencies automatically.

$ plandb task insert --after t-design --before t-impl --title "Add validation layer"

Options

FlagTypeDescription
--afterstringUpstream task (new task depends on this). Required.
--beforestringDownstream task (will depend on new task). Rewires the after-to-before edge.
--titlestringTitle of the new task. Required.
--descriptionstringDescription for the new task.
--projectstringProject ID

Example

# Before: t-design --> t-impl
$ plandb task insert --after t-design --before t-impl --title "Add validation"
# After:  t-design --> t-a3f7 "Add validation" --> t-impl

plandb task amend

Prepend context to a task's description. Use this to annotate future tasks with discoveries made during current work.

$ plandb task amend t-impl --prepend "NOTE: Use JWT, not sessions"

Arguments

NameRequiredDescription
<TASK_ID>YesTask ID to amend

Options

FlagTypeDescription
--prependstringText to prepend to the task's description. Required.

plandb task pivot

Replace a task's pending subtree with new tasks from JSON or YAML. Keeps completed subtasks by default with --keep-done.

$ plandb task pivot t-auth --file new-plan.yaml --keep-done

Arguments

NameRequiredDescription
<PARENT_ID>YesParent task whose subtree will be replaced

Options

FlagTypeDescription
--subtasksstringNew subtasks as JSON array: [{"title":"...","description":"..."}]
--filepathYAML file with new subtasks (alternative to --subtasks)
--keep-doneflagKeep already-completed subtasks (only replace pending/ready ones)

plandb task split

Split one task into multiple sub-tasks. Same as plandb split.

Server & Integration

Start MCP server (stdio JSON-RPC) for Claude Code, Cursor, and Windsurf integration.

Environment Variables

VariableDefaultDescription
PLANDB_AGENT default Default agent ID. Avoids passing --agent on every command. Set per-agent in multi-agent setups.
PLANDB_DB .plandb.db Path to SQLite database. Overrides --db flag. Useful for sharing a DB across directories.

Example: Multi-Agent Setup

# Terminal 1 — Agent A
$ export PLANDB_AGENT=agent-a
$ export PLANDB_DB=/shared/project.plandb.db
$ plandb go      # Claims as agent-a

# Terminal 2 — Agent B
$ export PLANDB_AGENT=agent-b
$ export PLANDB_DB=/shared/project.plandb.db
$ plandb go      # Claims a different task as agent-b

Task States

Every task follows this lifecycle:

pending  ---(deps satisfied)--->  ready  ---(go/claim)--->  claimed  ---(start)--->  running
                                                                                           |
                                                                               +-----------+-----------+
                                                                               |                       |
                                                                         done                  failed
StateMeaningTransitions To
pendingWaiting for upstream dependencies to completeready (automatic)
readyAll dependencies satisfied, available for claimingclaimed
claimedReserved by an agent, not yet startedrunning
runningActively being worked ondone, failed, paused
doneCompleted successfully (result data available)terminal
failedFailed with an error messageready (retry)

Note: plandb go combines claim + start in one atomic operation, transitioning directly from ready to running.

Dependency Types

TypeBlocks?Data Flow?Use When
feeds_into Yes Yes Task B needs Task A's result data (e.g., anatomy output feeds into planner)
blocks Yes No Task B must wait for A but doesn't use A's result (ordering only)
suggests No No Nice to have A done first, but B can proceed without it (soft dependency)

Specifying Dependencies

# Default: feeds_into
$ plandb add "Implement" --dep t-design

# Explicit type
$ plandb add "Implement" --dep t-design:feeds_into
$ plandb add "Deploy" --dep t-tests:blocks
$ plandb add "Docs" --dep t-impl:suggests

# Multiple dependencies
$ plandb add "Integration tests" --dep t-api --dep t-db

Task Kinds

KindDescription
genericDefault. Any task that doesn't fit a specific kind.
codeImplementation tasks: writing, modifying, or refactoring code.
researchInvestigation, exploration, or information gathering.
reviewCode review, design review, or security audit.
testWriting or running tests.
shellShell commands, infrastructure, or deployment tasks.

Use --kind when creating tasks to enable filtering with plandb list --kind code.
