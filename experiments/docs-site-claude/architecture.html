Architecture

PlanDB models work as a compound graph — two orthogonal structures layered on the same set of tasks. Understanding this model is the key to using PlanDB effectively.

Overview

Most task trackers give you either a hierarchy (nested folders, subtasks) or a dependency graph (Gantt charts, DAGs) — but not both. PlanDB gives you both, independently, on the same set of tasks.

The two structures are:

  • Place Graph (containment) — where a task lives. Projects contain tasks; tasks contain subtasks. A tree.
  • Link Graph (dependencies) — what a task needs before it can start. A directed acyclic graph (DAG) of data-flow and ordering edges.

Together they form a compound graph: the Place Graph defines scope and decomposition; the Link Graph defines execution order and data flow. They are independent — restructuring one does not break the other. 

                   COMPOUND GRAPH
    ┌─────────────────────┬─────────────────────────┐
    │   Place Graph        │   Link Graph             │
    │   (containment)     │   (dependencies)         │
    │                     │                          │
    │   project           │   task-A ──feeds──▶ task-C│
    │   ├── task-A        │       │                  │
    │   ├── task-B        │       └──blocks──▶ task-D│
    │   │   ├── task-D    │                          │
    │   │   └── task-E    │   task-B ──feeds──▶ task-E│
    │   └── task-C        │                          │
    │                     │                          │
    │   Tree structure     │   DAG structure           │
    └─────────────────────┴─────────────────────────┘

Place Graph (Containment)

The Place Graph is a tree that defines where tasks live. Every task belongs to exactly one parent — either a project or another task. This gives you hierarchical decomposition: a large task can be broken into subtasks, and those subtasks can be broken down further. 

  my-project                     ← project (root)
  ├── t-intake                    ← top-level task
  ├── t-analysis                  ← top-level task
  │   ├── t-ip-clauses            ← subtask of t-analysis
  │   ├── t-liability             ← subtask of t-analysis
  │   └── t-non-compete           ← subtask of t-analysis
  ├── t-review                    ← top-level task
  └── t-report                    ← top-level task

Scoping with plandb use

The plandb use command changes your working scope to any node in the Place Graph. All subsequent commands operate within that scope — plandb list shows only children, plandb add creates tasks inside it, and plandb status reports on that subtree. 

$ plandb use t-analysis
Scope set to t-analysis "Analysis phase"

$ plandb list
  t-ip-clauses    ready    "IP clause analysis"
  t-liability     ready    "Liability analysis"
  t-non-compete   ready    "Non-compete analysis"

$ plandb use --root
Scope reset to project root

This lets you zoom into a composite task, work on its children, and zoom back out — without losing track of the larger plan.

Link Graph (Dependencies)

The Link Graph is a DAG (directed acyclic graph) that defines execution order and data flow between tasks. Dependencies are edges between tasks — they determine which tasks are ready to execute.

A task becomes ready when all of its blocking dependencies (those of type feeds_into or blocks) are done. 

  t-intake ──feeds_into──▶ t-analysis ──feeds_into──▶ t-review
      │                                               │
      │                                               ▼
      └────────feeds_into──▶ t-report ◀──blocks──────┘

  t-research ···suggests···▷ t-analysis
  (soft dep: does not block)

Dependency Types

Type Blocks? Data flow? Use when
feeds_into Yes Yes Task B needs the output of Task A (e.g., anatomy result feeds into planner)
blocks Yes No Task B must wait for Task A, but doesn't use A's result (ordering constraint)
suggests No No Nice to have A done first, but B can proceed without it (soft/optional) 
$ plandb add --title "Run analysis" --dep t-intake:feeds_into
$ plandb add --title "Generate report" --dep t-analysis:feeds_into --dep t-review:blocks

Orthogonality

The Place Graph and Link Graph are independent. A task's position in the containment hierarchy says nothing about its dependencies, and vice versa. This is the key design insight that makes PlanDB's model powerful. 

  Place Graph (who owns it)          Link Graph (what it needs)

  project                           t-intake ──▶ t-ip-clauses
  ├── t-intake                      t-intake ──▶ t-liability
  ├── t-analysis                    t-intake ──▶ t-non-compete
  │   ├── t-ip-clauses              t-ip-clauses ──┐
  │   ├── t-liability               t-liability ───┼──▶ t-review
  │   └── t-non-compete             t-non-compete ─┘
  ├── t-review
  └── t-report                      t-review ──▶ t-report

  t-ip-clauses is a CHILD of t-analysis (place)
  but DEPENDS ON t-intake (link) — a sibling of its parent.
  These two relationships are orthogonal.

Why does this matter?

  • Decomposition is free. You can split a task into subtasks without rewiring dependencies. The subtasks inherit the parent's place but can depend on anything in the graph.
  • Dependencies are composable. You can add a dependency between any two tasks regardless of where they sit in the hierarchy. No need to restructure the tree to express a cross-cutting constraint.
  • Refactoring is safe. Moving a task to a different parent (re-parenting) does not change its dependencies. Changing dependencies does not move it in the hierarchy.

Task Lifecycle

Every task moves through a state machine. The transitions are enforced by PlanDB — you cannot skip states or move backward (except through explicit replan operations). 


          ┌──────────────────────────────────────────────────────┐
          │                  TASK STATE MACHINE                  │
          └──────────────────────────────────────────────────────┘

  deps unmet       deps met        agent claims     agent starts
 ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
 │          │   │          │   │          │   │          │
 │ PENDING  │──▶│  READY   │──▶│ CLAIMED  │──▶│ RUNNING  │
 │          │   │          │   │          │   │          │
 └──────────┘   └──────────┘   └──────────┘   └─────┬────┘
                                                     │
                                           ┌─────────┴─────────┐
                                           │                   │
                                           ▼                   ▼
                                     ┌──────────┐       ┌──────────┐
                                     │          │       │          │
                                     │   DONE   │       │  FAILED  │
                                     │          │       │          │
                                     └──────────┘       └──────────┘

Task States

State Badge Meaning
pending pending Blocked — one or more feeds_into or blocks dependencies are not yet done
ready ready All blocking dependencies satisfied — available for an agent to claim
claimed claimed An agent has atomically claimed this task but has not started execution
running running Agent is actively working on the task
done done Completed successfully — result stored, dependents may become ready
failed failed Failed — error recorded, dependents remain blocked

Automatic State Transitions

PlanDB automatically promotes pending tasks to ready whenever a dependency completes. When you run plandb done t-intake, PlanDB walks the Link Graph and promotes every task whose blocking dependencies are now all satisfied.

Multi-Agent Coordination

PlanDB is designed for parallel execution by multiple agents. The coordination mechanism is simple and race-free: atomic claiming via SQLite transactions.

How It Works

When an agent runs plandb go, PlanDB executes a single SQL transaction that:

  1. Finds the highest-priority ready task
  2. Sets its state to claimed and records the agent name
  3. Returns the task to the agent

Because SQLite serializes writes, two agents running plandb go simultaneously will never claim the same task. No locks, no distributed consensus — just SQLite's built-in transaction isolation.

Agent Identity

Agents identify themselves via the PLANDB_AGENT environment variable. This is used for tracking which agent owns which task, and for filtering: 

$ PLANDB_AGENT=worker-1 plandb go
Claimed t-intake "Implement intake classifier"

$ PLANDB_AGENT=worker-2 plandb go
Claimed t-analysis "Run analysis phase"

# Both agents work in parallel, no conflicts

$ plandb list --status running
  t-intake     running  worker-1  "Implement intake classifier"
  t-analysis   running  worker-2  "Run analysis phase"

Task Kinds

Each task has a kind that indicates the type of work. This is metadata for filtering and reporting — PlanDB does not enforce different behavior per kind.

Kind Typical use
codeImplementation work — writing or modifying source code
researchInvestigation, reading docs, exploring options
reviewCode review, architecture review, security audit
testWriting or running tests
shellRunning shell commands, deployments, infrastructure
genericDefault — anything that does not fit the above

Recursive Decomposition

Real-world tasks are often too large to complete in a single step. PlanDB supports recursive decomposition — splitting a running task into subtasks, zooming into the subtree, completing the children, and bubbling results back up.

The Split → Zoom → Work → Return Cycle

  1. SPLIT                           2. ZOOM
  Agent discovers t-analysis         Agent scopes into t-analysis
  is too large. Splits it:           to work on children:

  project                            t-analysis  ← current scope
  ├── t-intake  done                 ├── t-ip-clauses    ready
  ├── t-analysis  running            ├── t-liability     ready
  │   ├── t-ip-clauses               └── t-non-compete   ready
  │   ├── t-liability
  │   └── t-non-compete
  ├── t-review  pending
  └── t-report  pending

  3. WORK                            4. RETURN
  Complete subtasks (possibly         All children done; parent
  in parallel across agents):         auto-completes. Dependents
                                      become ready:
  t-analysis
  ├── t-ip-clauses    done           project
  ├── t-liability     done           ├── t-intake     done
  └── t-non-compete   done           ├── t-analysis   done
                                      ├── t-review     ready  ← unblocked!
                                      └── t-report     pending
# Split a running task into subtasks
$ plandb task split t-analysis --into '[
    {"title": "IP clause analysis"},
    {"title": "Liability analysis"},
    {"title": "Non-compete analysis"}
  ]'

# Zoom into the subtree
$ plandb use t-analysis

# Work on children (or dispatch to parallel agents)
$ PLANDB_AGENT=worker-1 plandb go
$ PLANDB_AGENT=worker-2 plandb go
$ PLANDB_AGENT=worker-3 plandb go

Plan Adaptation

Plans change. PlanDB provides operations to modify the task graph mid-flight while preserving its structural invariants (no cycles in the Link Graph, single parent in the Place Graph).

Adaptation Operations

Operation Command What it does
Insert plandb task insert Add a new task between two existing tasks, automatically rewiring dependency edges
Amend plandb task amend Add context to a future task's description based on discoveries from earlier tasks
Split plandb task split Decompose a task into subtasks (Place Graph change, Link Graph preserved)
Replan plandb task replan Cancel a pending subtree and replace it with a new plan from a YAML file

Insert: Adding a Step Mid-Flight

When you discover a missing step between two tasks, insert splices it in and rewires the dependency edges: 

  Before:  t-intake ──feeds──▶ t-analysis

  $ plandb task insert --after t-intake --before t-analysis \
      --title "Jurisdiction detection"

  After:   t-intake ──feeds──▶ t-jurisdiction ──feeds──▶ t-analysis

Amend: Annotating Future Tasks

As earlier tasks produce insights, annotate downstream tasks so future agents (or future sessions) have the context they need: 

$ plandb task amend t-review --prepend \
    "NOTE: Section 14 has a survival clause that affects termination analysis"

Replan: Replacing a Subtree

When a section of the plan is no longer valid, cancel it and load a replacement: 

$ plandb task replan t-review --file revised-review.yaml
Cancelled 3 pending tasks under t-review
Created 5 new tasks from revised-review.yaml
Dependency edges rewired

Invariants Preserved

All adaptation operations maintain two invariants:

  • No cycles in the Link Graph — PlanDB rejects any dependency that would create a cycle
  • Single parent in the Place Graph — every task has exactly one parent (project or task)

These invariants ensure that the graph always has a well-defined topological order (so plandb go always knows what to execute next) and a well-defined scope hierarchy (so plandb use always has an unambiguous subtree).
