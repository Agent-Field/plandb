Architecture

PlanDB is built on the concept of a compound graph. This is not just a tree or just a DAG, but two independent structures composed together.

The Place Graph (Containment)

The place graph defines how tasks are nested. Just like files and folders in a filesystem, tasks in PlanDB can contain subtasks to any depth.

  • Tasks at the top level have depth 0.
  • Subtasks can be added to any existing task.
  • Recursive completion: A parent task is automatically completed when all of its children are done.

The Link Graph (Dependencies)

The link graph defines the execution order. It is an orthogonal structure that is completely independent of the containment tree.

  • Any task can depend on any other task, regardless of their position in the tree.
  • Dependencies can cross branch boundaries—a subtask at depth 3 can depend on a task at depth 0 in a different branch.
  • This allows for complex build-like dependency flows within a clean organizational hierarchy.

Composition

When combined, these structures allow for sophisticated workflows. An AI agent can look at the containment tree to understand the scope of a task, and follow the link graph to determine what is ready to be worked on.

$ plandb status --full
# Shows both the tree and the dependency edges!
