Architecture

PlanDB models work as a compound graph. One structure captures containment and decomposition. Another captures execution dependencies. They overlap on the same tasks, but they answer different questions.

The place graph

The place graph is the containment structure. It tells you which tasks belong inside which larger tasks. That is how PlanDB can represent coarse work items that later decompose into smaller subtasks without losing the parent-child relationship.

In practical terms, the place graph answers: “What is part of this task?” It is about hierarchy, scope, and recursive decomposition.

The link graph

The link graph is the dependency structure. It tells you which tasks must finish before others can begin. This is what powers dependency-aware claiming with plandb go.

The link graph answers: “What must happen before this task can run?” It is about ordering, readiness, and execution flow.

Two orthogonal structures on the same tasks

The same task nodes participate in both graphs. That is the key architectural move: hierarchy and dependency stay separate instead of being collapsed into a single structure.

Why orthogonality matters

A flat todo list loses hierarchy. A simple DAG captures order but not containment. If you force both concerns into one graph, decomposition often distorts execution semantics.

PlanDB avoids that by keeping the place graph and link graph orthogonal. A task can contain subtasks without implying that every child depends on every sibling. Dependency edges stay explicit and precise.

Recursive decomposition

Recursive decomposition means a task can begin as a coarse work item, then split into smaller subtasks when more information becomes available. The parent task remains the conceptual container, while the children carry the executable detail.

This is especially useful for AI agents. An agent can claim a broad task, realize it needs more structure, split it, and let the graph evolve instead of discarding the original plan.

How this maps to PlanDB workflows

This separation is why PlanDB can support both hierarchical planning and dependency-safe execution. Agents can navigate the work tree while the scheduler still reasons about readiness using explicit edges.
