Getting Started

PlanDB works best when each task is a complete work order: short title, detailed description, and explicit dependencies. This walkthrough takes a small release plan from initialization through execution and graph inspection.

Initialize a project

Start by creating a named project. PlanDB stores the graph in SQLite, so the database becomes the shared state for every agent working on the same effort. 

$ plandb init "agent-launch"
created p-launch (agent-launch)

Use the project name to describe the actual initiative, not just a ticket. That makes status output legible when several efforts share one database.

Add tasks with dependencies

Add tasks in dependency order so downstream tasks can point to task IDs that already exist. Descriptions should be detailed enough for another agent to execute the task later with no hidden context. 

$ plandb add "Design launch checklist" --as checklist --kind research --description "Define the release checklist, required owners, and dependencies for the agent launch."
created task t-checklist

$ plandb add "Draft announcement" --as announce --kind code --dep t-checklist --description "Write the launch announcement copy using the final checklist and release scope."
created task t-announce

$ plandb add "Publish docs" --as docs --kind code --dep t-announce --description "Update docs site navigation, release notes, and rollout instructions after announcement copy is approved."
created task t-docs

In this example, `t-announce` cannot begin until `t-checklist` is done, and `t-docs` waits for the announcement task. The graph now encodes execution order instead of relying on memory.

Claim the next ready task

Use `plandb go` to claim and start the next ready task. If several agents are connected to the same project, PlanDB prevents them from claiming the same node. 

$ plandb go
→ t-checklist "Design launch checklist" [1/3 · 1 ready]

This is the preferred entry point for execution. The command only returns tasks whose dependencies are already complete.

Complete and continue with done --next

When the current task is finished, complete it and immediately claim the next ready task. This keeps the execution loop tight. 

$ plandb done --next
✓ t-checklist done → claimed t-announce "Draft announcement"

Use `--result` when a downstream task needs structured handoff data, and `--files` when you want PlanDB to record which files changed during the task.

Split work when a task becomes too broad

If a task turns out to be larger than expected, split it while working. Use a comma-separated list for parallel parts or a `>` chain when the new subtasks still need internal sequencing. 

$ plandb split --into "Write email draft, Review with legal, Schedule send"

$ plandb split --into "Draft changelog > QA release notes > Publish release post"

Split when you discover new structure. Do not wait until the task is blocked or ambiguous. Smaller nodes make it easier for another agent to pick up a piece of work safely.

Inspect graph progress with status

Use `plandb status --detail` to see the live task breakdown and dependency tree. This is the fastest way to check what is done, ready, running, or still blocked. 

$ plandb status --detail
p-launch agent-launch: 1/3 done (33%)
✓ t-checklist Design launch checklist
◉ t-announce Draft announcement
○ t-docs Publish docs

The graph state is now inspectable by anyone, human or agent. There is no need to reconstruct the plan from prose updates alone.

Why descriptions matter

A task title is only a label. The description is the executable contract. It should name the files, the intended output, the acceptance criteria, and any constraints that would otherwise stay in someone’s head.

That matters even more with AI agents. A future agent should be able to run `plandb show t-announce`, read the description, and start work without external chat context.
