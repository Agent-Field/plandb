What PlanDB is

PlanDB is the task graph primitive behind agent orchestration. It stores work in SQLite, enforces dependencies, and gives agents a simple operational loop: claim the next ready task, perform the work, then complete and continue.

Shared state for agent work

Every task lives in a persistent graph with explicit IDs, descriptions, dependencies, status, and ownership. An agent can stop, resume, or hand off work without reconstructing context from chat history.

Dependency-aware execution

`plandb go` only claims tasks whose upstream dependencies are complete. That gives you safe parallelism by default instead of optimistic concurrency and manual coordination.

Recursive decomposition

Work starts coarse, then splits as you learn more. Subtasks can live inside parent tasks while dependency edges continue to express execution order across the broader graph.

Why it exists

AI agents need more than a flat todo list. They need a durable model of work that distinguishes structure from sequencing: what belongs inside what, and what must happen before what.

PlanDB exists to represent both at once. The place graph models containment and decomposition. The link graph models dependencies. Keeping those structures orthogonal avoids the usual failure mode where hierarchy and execution order get conflated.

The result is a compound graph that matches how agents actually work: break tasks down, coordinate across specialists, and continue from shared project state instead of ephemeral prompts.

That loop is the operational center: initialize a project, encode work with dependencies, claim what is ready, and complete tasks while automatically advancing.

Why it fits AI agent orchestration

PlanDB makes agent collaboration explicit. Tasks are not just titles; they are instructions. Dependencies are not conventions; they are enforced edges.

Descriptions as work orders

Each task description can specify exact files, acceptance criteria, and constraints so a different agent can pick it up from `plandb show` and work independently.

Safe multi-agent claiming

Multiple agents can run the same workflow against one database. Ready-state gating prevents duplicate ownership and keeps execution aligned with the actual graph.

Adaptive decomposition

When a task grows during execution, split it into parallel parts or a chained sequence instead of rewriting the plan from scratch.

Operational clarity

`status`, `show`, and the task-level tools make the project state inspectable at any time. The graph becomes the source of truth for both humans and agents.
